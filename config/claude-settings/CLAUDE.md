# 開発時に気をつけること

- CLAUDE.md は基本的に日本語で記載すること
- 基本的に日本語でやり取りすること

## 開始前の確認

- `git pull`を実行して、リモートの変更をローカルに反映すること
- Node.js のプロジェクトの場合、最新の依存関係をインストールすること

## 開発用ブランチの作成

- ブランチの作成元はプロジェクト毎に異なるので、都度確認すること。
- ブランチ名は以下のいずれかに従うこと
  - 修正対象の JIRA 番号が指定されている場合、JIRA 番号を prefix にすること
  - 例: `XXXX-1234`が指定されている場合、`XXXX-1234-修正内容`とする
  - GitHub や GitLab の Issue 番号が指定されている場合、`<番号>-修正内容`とする
    - 例: `1234`が指定されている場合、`1234-修正内容`とする
  - 上記以外の場合は修正対象の JIRA や issue がないかを確認し、なければ修正内容を簡潔に表すブランチ名にする

## コミットメッセージ

- コミットメッセージに `Co-Authored-By` の行は付けないこと
- プロジェクトで指定がある場合はそれに従うこと
- 特に指定がない場合は以下のルールに従うこと
  - コミットメッセージは日本語で書くこと
  - conventional commit の形式に従うこと
  - `feat: 機能追加` / `fix: バグ修正` / `docs: ドキュメントの変更` / `style: フォーマットの変更` / `refactor: リファクタリング` / `test: テストコードの追加・修正` / `chore: その他の変更`

## ルールの更新

- 開発者からの指示で「常に」や「必ず」などの表現がある場合は、開発者に確認したうえでルールを更新すること
- ルールの更新はプロジェクトに関係するものはプロジェクト内の CLAUDE.md に記載し、開発者個人のルールは本ファイルに記載すること

## 計画作成のルール

### plan モードへの移行(原則必須)

- 開発・対応の指示を受けた場合、まず plan モードに入って計画を検討すること
- auto モードであっても本ルールは適用される(auto モードは「実装フェーズの確認頻度を下げる」ものであって、「最初の方針合意」を省略するものではない)
- ただし以下のスキップ条件に該当する場合は plan モードを経由しなくて良い:
  - typo・lint エラー解消・1行レベルの修正
  - 既存パターンへの完全追従(類似コードをコピーして1箇所変えるだけ)
  - ユーザーが明示的に「計画不要」「すぐ実装して」と指示した場合

### 計画ファイルの出力(必須)

- 計画は以下の場所・命名規則で出力すること(個人用、リポジトリには commit しない):
  - 保存先: `~/.claude/projects/<repo-slug>/plans/<YYYY-MM>/`
    - `<repo-slug>` は Claude Code がプロジェクトディレクトリから自動生成するスラッグ
    - `<YYYY-MM>` は計画書を書いた月(例: `2026-04`)
    - フォルダがなければ `mkdir -p` で先行作成
    - 例: `~/.claude/projects/-data-git-webconnect-material-registration/plans/2026-04/`
  - 命名: `<YYYYMMDD>_<JIRA番号>_plan.md` (例: `20260427_KKGMRDEV-15175_plan.md`)
  - JIRA 番号がない場合は `<YYYYMMDD>_<内容を表す短い英名>_plan.md`
- ブラッシュアップ後は `_v2`, `_v3`, `_final` を末尾に付けて履歴を残すこと(上書きしない)
- 月またぎの作業でも `_v2` 以降は計画開始月のフォルダに置き続ける(履歴の連続性を優先)
- なお、plan モードのスクラッチパッド(`~/.claude/plans/<ランダム名>.md`)とは別物。本ルールは**永続的な開発計画書**を指す
- 計画書には以下のセクションを含めること:
  - **Context**: 変更の背景・動機・期待される成果
  - **影響範囲**: 修正対象の主要ファイルパス一覧
  - **既存利用要素**: 再利用する既存関数・ユーティリティ(ファイルパス付き)
  - **実装手順**: 段階的な作業ステップ
  - **検証方法**: どうテストするか(コマンド・MCPツール・テスト名を具体的に)
  - **Pre-mortem**: 想定される失敗パターンとその予防策

### 計画の質を上げる手順(規模別)

#### 小規模タスク (1ファイル / 50行未満の変更)

- plan モードでの単発計画で十分
- ブラッシュアップは原則不要

#### 中規模タスク (2〜3ファイル / 既存機能の拡張)

1. **調査**: Read/Grep で類似の既存実装を3つ以上確認
2. **ドラフト**: `superpowers:brainstorming` → `superpowers:writing-plans` で初版を作成し `_v1` で保存
3. **レビュー**: `claude-code-plan-review` skill で4軸(アーキテクチャ / コード品質 / テスト / パフォーマンス)レビューを実施
4. **修正版を `_v2` で保存** してユーザー承認を取る

#### 大規模タスク (4ファイル以上 / 新規機能 / スキーマ変更 / アーキテクチャ変更)

1. **地形調査**: `feature-dev:code-explorer` agent で既存実装の構造・パターンをマッピング
2. **ドラフト**: `superpowers:brainstorming` → `superpowers:writing-plans` で初版を `_v1` 保存
3. **並列観点レビュー**(複数 agent を1メッセージ内で並列起動):
   - `claude-code-plan-review` (汎用4軸)
   - 該当する `reviewer-*` agent (`reviewer-multi-tenant` / `reviewer-sql-prisma` / `reviewer-external-api-contract` / `reviewer-project-aware`)
4. **敵対的検証**: 別 Agent を独立に起動し「この計画の最大の弱点は?」「失敗するとしたら最初に壊れるのはどこ?」を批判的に検証させる
5. **Pre-mortem セクションの強化**: 「リリース後3週間で起きうるインシデント」を列挙し、各インシデントへの予防策を計画に組み込む
6. **最終版を `_final` で保存** してユーザー承認を取る

### 実装フェーズへの移行

- 計画書がユーザーに承認されたら ExitPlanMode で plan モードを抜けて実装を開始する
- auto モード中であれば、承認後の実装は連続実行して良い(細かい確認の差し込みは最小化)
- 実装中に計画書から外れる判断が必要になった場合は、その時点で計画書を `_v(N+1)` として更新するか、ユーザーに確認すること

## 実装後の学びログ(Obsidian)

### トリガー(常時自動)

- plan モードを経由した実装が完了したら、**規模を問わず必ず**学びログを Obsidian に出力すること
- plan モードをスキップした小タスク(typo 修正等)では学びログ作成は不要
- ユーザーへの確認は不要(自動で書く)

### 保存先・命名

- **保存先**: `/mnt/c/Users/hiroaki_hazama/Documents/obsidian_vault/main/18_claude_learnings/<YYYY-MM>/`
  - Windows 側パス(参考): `C:\Users\hiroaki_hazama\Documents\obsidian_vault\main\18_claude_learnings\<YYYY-MM>\`
  - `<YYYY-MM>` は学びログを書いた月(例: `2026-04`)
  - フォルダがなければ `mkdir -p` で先行作成
- **命名規則**: `<YYYYMMDD>_<日本語の読みやすいタイトル>.md`
  - 例: `2026-04/20260427_JTH バッチ partial run の発見.md`
  - JIRA 番号はファイル名に含めず frontmatter に記載

### auto memory との関係

- `~/.claude/projects/<repo>/memory/` の auto memory は既存の自動挙動のまま稼働
- Obsidian 学びログは別系統として独立運用する(両者を強制的に同期しない)
- 同じ事実が両方に書かれることは許容する(読者が異なるため)
- ただし、Obsidian の学びの中に「これは Claude が次回以降覚えておくべき」と判断されるものがあれば、auto memory にも昇格させてよい

## 開発中の原則

- 不明なロジックや仕様は推測せず、確認事項として報告すること

## コーディング規約

- **型定義**: `any`は禁止。適切な型/インターフェースを定義する
- **型キャスト**: `as`によるキャストは最小限にし、型ガードを使う
- **クラス**: 200行以下、publicメソッド10個以下
- **関数**: 最大20行以内
- **テスト命名**: `describe('{クラス名}')` → `it('～の場合、～すべき')`
- **コメント**: 「なぜ」を説明するコメントのみ書く。「何を」しているかはコードで表現する

## SQL規約

参照: http://gitlab.fdev/-/snippets/21

- 予約語・関数は小文字（`select`, `from`, `where`, `insert into`）
- テーブル名・カラム名はsnake_case
- 比較演算子: `!=`ではなく`<>`を使う
- カンマは前置（前カンマ）。カンマ後にスペースは入れない
- `and`/`or`は行頭に置く
- `join`は`from`から一段下げる
- サブクエリはなるべく避け`with`（CTE）で書く
- テーブルエイリアス: `as`不要 / カラムエイリアス: `as`必須
- `case`には必ず`else`を書く
- `right outer join`は使わない
- `union`ではなく`union all`を使う。`union`を使う場合は理由をコメントに書く
- `distinct`/`first`はできるだけ避ける。使う場合は理由をコメントに書く
- セミコロンは最後の行で改行して行頭に置く

```sql
-- フォーマット例
select
	col_a
	,col_b
	,case
		when col_c < 10 then
			1
		else
			0
	end as flag
from
	table_a ta
	inner join table_b tb using(id)
where
	ta.status <> 'deleted'
	and tb.value > 0
;
```
